# 運用手順書 (Operations)

本プロジェクトを長期にわたって安全かつスムーズに運用するための手順・ルールです。

---

## 1. 記事追加手順
記事を新規に執筆・掲載する際は、以下のステップで進めます。

### ① Markdownファイルの作成
`src/content/posts/` ディレクトリ内に、英語の小文字とハイフンを組み合わせたファイル名（例：`care-website-essentials.md`）で新規ファイルを作成します。

### ② フロントマターの定義
ファイルの最上部に、以下の項目を正確に定義します。
```yaml
---
title: "記事のタイトル（SEOキーワードを意識）"
description: "検索結果や一覧に表示される100文字程度の概要"
publishedAt: "2026-07-07"              # YYYY-MM-DD 形式
category: "カテゴリ名（介護, Web, 営業, AI, 情報設計 のいずれか）"
tags: ["タグ1", "タグ2"]                 # 既存タグに寄せる
author: "H3 Incover"
eyecatch: ""                           # 個別アイキャッチがある場合のみURL指定、ない場合は空欄
recommend: false                       # 推奨記事としてトップ等に固定する場合は true
relatedPosts: ["slug-1", "slug-2"]     # 関連記事のスラッグを最大3つ指定
---
```

### ③ 本文の執筆とフォーマット統一
H3 Incover INFO 標準の記事構成ルールに従い、マークダウンにて執筆します。

---

## 2. 記事の表示構成（標準フォーマット）
すべての記事は、読者の読みやすさと回遊性を均一にするため、以下の順番で構成します。

1. **タイトル** (メタデータからJSXで自動表示)
2. **導入** (本文の最初段落: 共感や日常の対話を交えた自然な入り口)
3. **目次** (JSXで自動挿入)
4. **シリーズ案内** (シリーズ連載の場合のみ: 薄いグレー背景のHTMLカード)
5. **本文** (Markdown H2/H3 構造)
6. **H3 Incoverの考え方** (黒背景のHTMLカード)
7. **まとめ** (全体の要約、およびブランド思想へ紐づく結び)
8. **あわせて読みたい** (公開済みの内部リンクのみを表示するHTMLカード)
9. **タグ** (JSXで自動表示)
10. **共有** (SNS共有ボタン: JSXで自動表示)
11. **CTA** (共通お問い合わせブロック: JSXで自動表示)
12. **関連記事** (自動アサインされたカード: JSXで自動表示)

## 3. レビューとGit運用手順

### ① ローカル表示確認
Git コミット前に必ずローカル開発サーバーを起動し、テキストやレイアウトが崩れていないか確認します。
```bash
# 開発サーバー起動
npm run dev
# 表示URL: http://localhost:3000/posts/[slug]
```

### ② ビルドテストの実施
静的ページの生成（SSG）でエラーが出ないかを検証するため、プッシュ前に必ずビルドテストを回してください。
```bash
npm run build
```

### ③ Gitコミット・プッシュ（本番反映）
レビューで承認された後、GitHubへプッシュします。
```bash
# 変更内容のステージング
git add .
# コミット（規約に準拠したメッセージ）
git commit -m "feat: Add care website absence problems article"
# プッシュ
git push origin main
```

## 4. 特殊アセットの管理手順

### 画像の追加
* **記事別のアイキャッチ**: 独自の画像を用意した場合は、`public/images/posts/` に格納し、フロントマターの `eyecatch` に `/images/posts/filename.png` のパスを指定します。
* **カテゴリデフォルト画像**: カテゴリ新設に伴ってデフォルト画像を定義する場合は、`public/images/categories/` 以下に配置します（ファイル名はカテゴリ名のアルファベット表記、小文字、アンダースコア推奨）。

### カテゴリの追加・変更
新しいカテゴリを追加する場合は、`src/components/ImageWithFallback.tsx` 内の `categoryFallbackMap` にカテゴリ名と対応するデフォルト画像名を追加してください。マップに定義されていないカテゴリは、自動的に `other.png` にフォールバックされます。

## 5. バックアップとトラブルシューティング

### バックアップ方針
* **コードとコンテンツ**: GitHub リモートリポジトリがマスターバックアップとなります。
* **アセット**: デザインの元データや元画像は、外部クラウドストレージ等で別途管理することを推奨します。

### トラブルシューティング: ビルド時のGoogle Fontsフェッチエラー
ビルド時（`npm run build`）に `Failed to fetch Noto Sans JP from Google Fonts.` というエラーが発生することがあります。これは通信環境による一時的な接続不良です。コードのエラーではないため、少し時間をおいてから再度ビルドを実行してください。
